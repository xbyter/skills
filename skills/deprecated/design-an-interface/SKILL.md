# 设计接口

基于"A Philosophy of Software Design"中的"Design It Twice"：你的第一个想法不太可能是最好的。生成多个截然不同的设计，然后进行比较。

## 工作流

### 1. 收集需求

设计前，了解：

- [ ] 这个模块解决什么问题？
- [ ] 调用者是谁？（其他模块、外部用户、测试）
- [ ] 关键操作有哪些？
- [ ] 有什么约束？（性能、兼容性、现有模式）
- [ ] 什么应该隐藏在内部 vs 暴露出来？

问："这个模块需要做什么？谁会使用它？"

### 2. 生成设计（并行 Sub-Agents）

使用 Task tool 同时生成 3+ 个 sub-agents。每个必须产生一个**截然不同**的方案。

```
每个 sub-agent 的提示模板：

为以下模块设计接口：[模块描述]

需求：[收集的需求]

此设计的约束：[为每个 agent 分配不同的约束]
- Agent 1: "最小化方法数量 - 目标最多 1-3 个方法"
- Agent 2: "最大化灵活性 - 支持多种用例"
- Agent 3: "为最常见的情况优化"
- Agent 4: "从 [特定范式/库] 中获取灵感"

输出格式：
1. 接口签名（类型/方法）
2. 使用示例（调用者如何使用它）
3. 此设计在内部隐藏了什么
4. 此方案的权衡
```

### 3. 呈现设计

每个设计展示：

1. **接口签名** - 类型、方法、参数
2. **使用示例** - 调用者在实践中实际如何使用
3. **隐藏了什么** - 保持在内部的复杂性

按顺序呈现设计，让用户在比较前能理解每种方案。

### 4. 比较设计

展示所有设计后，在以下方面进行比较：

- **接口简洁性**：更少的方法，更简单的参数
- **通用 vs 专用**：灵活性 vs 专注性
- **实现效率**：形状是否允许高效的内部实现？
- **深度**：小接口隐藏显著复杂性（好）vs 大接口配薄实现（坏）
- **正确使用的便捷性** vs **误用的便捷性**

以散文而非表格讨论权衡。突出设计差异最大的地方。

### 5. 综合

通常最好的设计融合了多个选项的洞察。问：

- "哪个设计最适合你的主要用例？"
- "其他设计中有什么元素值得融入？"

## 评估标准

来自"A Philosophy of Software Design"：

**接口简洁性**：更少的方法，更简单的参数 = 更容易学习和正确使用。

**通用性**：能处理未来的用例而无需更改。但要警惕过度泛化。

**实现效率**：接口形状是否允许高效实现？还是迫使笨拙的内部结构？

**深度**：小接口隐藏显著复杂性 = 深模块（好）。大接口配薄实现 = 浅模块（避免）。

## 反模式

- 不要让 sub-agents 产生相似的设计 - 强制截然不同
- 不要跳过比较 - 价值在于对比
- 不要实现 - 这纯粹是关于接口形状
- 不要基于实现工作量来评估
