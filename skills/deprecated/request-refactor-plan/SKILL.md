---
name: 请求重构计划
description: 通过用户访谈创建详细的重构计划，包含小步提交，然后将其作为 GitHub issue 归档。当用户想要规划重构、创建重构 RFC 或将重构拆分为安全的增量步骤时使用。
---

当用户想要创建重构请求时，此 skill 将被调用。你应该完成以下步骤。如果认为某些步骤不必要，可以跳过。

1. 请用户详细描述他们想解决的问题以及可能的解决方案想法。

2. 探索仓库以验证他们的断言并了解代码库的当前状态。

3. 询问他们是否考虑过其他选项，并向他们展示其他选项。

4. 就实现方案采访用户。要非常详细和深入。

5. 确定实现的确切范围。确定你计划改什么和不改什么。

6. 在代码库中查找此区域的测试覆盖率。如果测试覆盖率不足，询问用户的测试计划。

7. 将实现拆分为小提交的计划。记住 Martin Fowler 的建议"使每个重构步骤尽可能小，这样你总是能看到程序在运行。"

8. 使用重构计划创建 GitHub issue。使用以下模板作为 issue 描述：

<refactor-plan-template>

## 问题陈述

开发者面临的问题，从开发者的角度。

## 解决方案

问题的解决方案，从开发者的角度。

## Commits

一个很长的、详细的实现计划。用简单英语编写，将实现拆分为尽可能小的 commits。每个 commit 应使代码库保持在工作状态。

## 决策文档

已做出的实现决策列表。可以包括：

- 将构建/修改的模块
- 将被修改的模块接口
- 开发者的技术澄清
- 架构决策
- Schema 变更
- API 合约
- 特定交互

不要包含具体的文件路径或代码片段。它们可能很快就会过时。

## 测试决策

已做出的测试决策列表。包括：

- 什么是好测试的描述（只测试外部行为，不测试实现细节）
- 哪些模块将被测试
- 测试的先例（即代码库中类似类型的测试）

## 超出范围

此重构超出范围的事项描述。

## 补充说明（可选）

关于重构的任何补充说明。

</refactor-plan-template>
