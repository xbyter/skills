# 测试驱动开发

## 哲学

**核心原则**：测试应通过公共接口验证行为，而非实现细节。代码可以完全改变；测试不应该。

**好测试**是集成风格的：它们通过公共 API 行使真实代码路径。它们描述系统_做什么_，而不是_怎么做_。好测试读起来像规范——"用户可以用有效购物车结账"准确告诉你存在什么能力。这些测试能经受重构，因为它们不关心内部结构。

**坏测试**与实现耦合。它们 mock 内部协作者、测试私有方法，或通过外部手段验证（如直接查询数据库而非使用接口）。警告信号：重构时测试失败，但行为没有改变。如果你重命名一个内部函数后测试失败，那些测试测的是实现，不是行为。

参见 [tests.md](tests.md) 获取示例和 [mocking.md](mocking.md) 获取 mock 指南。

## 反模式：水平切片

**不要先写所有测试，再写所有实现。** 这是"水平切片"——把 RED 当作"写所有测试"，把 GREEN 当作"写所有代码"。

这会产生**垃圾测试**：

- 批量编写的测试测试的是_想象的_行为，不是_实际的_行为
- 最终测试的是事物的_形状_（数据结构、函数签名）而非面向用户的行为
- 测试对真实变化变得不敏感——行为出错时通过，行为正常时失败
- 你在车灯前跑得太远，在理解实现之前就承诺了测试结构

**正确方法**：通过 tracer bullet 进行垂直切片。一个测试 → 一个实现 → 重复。每个测试回应你从上一个循环中学到的东西。因为你刚写了代码，你确切知道什么行为重要以及如何验证它。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流

### 1. 规划

探索代码库时，使用项目的领域词汇表，使测试名称和接口词汇与项目语言匹配，并尊重你触及区域的 ADR。

在编写任何代码之前：

- [ ] 与用户确认需要什么接口变更
- [ ] 与用户确认要测试哪些行为（确定优先级）
- [ ] 识别 [deep modules](deep-modules.md) 的机会（小接口，深实现）
- [ ] 为[可测试性](interface-design.md)设计接口
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获取用户对计划的批准

问："公共接口应该是什么样的？哪些行为最重要需要测试？"

**你无法测试所有东西。** 与用户确认哪些行为最重要。将测试精力集中在关键路径和复杂逻辑上，而非每个可能的边界情况。

### 2. Tracer Bullet

编写一个测试来确认关于系统的一件事：

```
RED:   为第一个行为写测试 → 测试失败
GREEN: 写最小代码使其通过 → 测试通过
```

这是你的 tracer bullet——证明路径端到端工作。

### 3. 增量循环

对于每个剩余行为：

```
RED:   写下一个测试 → 失败
GREEN: 最小代码使其通过 → 通过
```

规则：

- 一次一个测试
- 只写足够让当前测试通过的代码
- 不要预期未来的测试
- 保持测试关注可观察行为

### 4. 重构

所有测试通过后，寻找[重构候选](refactoring.md)：

- [ ] 提取重复
- [ ] 深化模块（将复杂性移到简单接口后面）
- [ ] 在自然的地方应用 SOLID 原则
- [ ] 考虑新代码揭示了现有代码的什么
- [ ] 每次重构步骤后运行测试

**永远不要在 RED 状态下重构。** 先到 GREEN。

## 每次循环的检查清单

```
[ ] 测试描述行为，而非实现
[ ] 测试仅使用公共接口
[ ] 测试能经受内部重构
[ ] 代码对这个测试来说是最小的
[ ] 没有添加投机性功能
```
