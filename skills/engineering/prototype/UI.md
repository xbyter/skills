# UI 原型

在单个路由上生成**多种截然不同的 UI 变体**，从浮动底部栏切换。用户在浏览器中翻转于变体之间，选择一个（或从每个中窃取部分），然后丢弃其余的。

如果问题关于逻辑/状态而非某东西看起来什么样 —— 错误的分支。使用 [LOGIC.md](LOGIC.md)。

## 何时这是正确的形状

- "这个页面应该长什么样？"
- "在提交之前我想看几个这个仪表板的选项。"
- "为设置屏幕尝试不同的布局。"
- 任何用户否则会花一天时间在脑海中三个模糊的模型之间选择的场景。

## 两种子形状 —— 强烈偏好子形状 A

当 UI 原型**紧靠应用的其余部分**时更容易评判 —— 真实的头部、真实的侧边栏、真实的数据、真实的密度。一个独立的一次性路由是真空：每个变体在隔离中看起来都好。默认选择子形状 A，只要有一个合理的现有页面来承载变体。仅当原型确实没有附近的家时才选择子形状 B。

### 子形状 A —— 调整现有页面（首选）

路由已存在。变体在**同一路由**上渲染，通过 `?variant=` URL search param 控制。现有的数据获取、参数和认证都保留 —— 只有渲染交换。这是默认选择；除非有特定理由不选。

如果原型是为尚未有页面的东西但*自然会在一个页面内*（仪表板的新部分、设置屏幕的新卡片、现有流程的新步骤）—— 那仍然是子形状 A。在宿主页面内挂载变体。

### 子形状 B —— 新页面（最后手段）

仅当被原型化的东西确实没有现有页面可嵌入时使用 —— 例如全新的顶级界面，或无法嵌入任何合理位置的流程。

创建**一次性路由**，遵循项目已有的路由约定 —— 不要发明新的顶层结构。命名使其明显是原型（例如在路径或文件名中包含 `prototype` 一词）。使用相同的 `?variant=` 模式。

在承诺子形状 B 之前，理智检查：真的没有现有页面可以嵌入吗？空路由隐藏了填充路由会暴露的设计问题。

在两种子形状中浮动底部栏是相同的。

## 流程

### 1. 陈述问题并选择 N

默认 **3 个变体**。超过 5 个就不再是截然不同而是噪音 —— 上限在那里。

用一行写下计划，放在原型的位置或文件顶部的注释中：

> "设置页面的三个变体，通过 `?variant=` 切换，在现有 `/settings` 路由上。"

无论用户是否在场推动都行。

### 2. 生成截然不同的变体

起草每个变体。坚持：

- 页面的目的和它可访问的数据。
- 项目的组件库/样式系统（TailwindCSS、shadcn、MUI、纯 CSS，无论什么）。
- 清晰的导出组件名，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须是**结构上不同的** —— 不同的布局、不同的信息层次、不同的主要可供性，而不仅仅是不同的颜色。三个稍微调整的卡片网格不是 UI 原型，是墙纸。如果两个草稿太相似，用明确的"不要使用卡片网格"指导重做一个。

### 3. 将它们连接起来

在路由上创建单个切换组件：

```tsx
// 伪代码 —— 适配项目的框架
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形状 A（现有页面）：保持所有现有数据获取在切换器之上；只有渲染的子树按变体变化。

对于子形状 B（新页面）：`/prototype/<name>` 下的一次性路由挂载相同的切换器。

### 4. 构建浮动切换器

一个小型固定位置栏在屏幕底部中央，有三个部分：

- **左箭头** —— 循环到上一个变体（环绕）。
- **变体标签** —— 显示当前变体键，如果变体导出了名称也显示。例如 `B — Sidebar layout`。
- **右箭头** —— 向前循环（环绕）。

行为：

- 点击箭头更新 URL search param（使用框架的路由 —— Next 上 `router.replace`，React Router 上 `navigate` 等）以便变体可分享且刷新稳定。
- 键盘：`←` 和 `→` 方向键也可循环。当 `<input>`、`<textarea>` 或 `[contenteditable]` 有焦点时不要拦截方向键。
- 与页面视觉上不同（例如高对比度药丸、微妙阴影），以便明显不是被评估设计的一部分。
- 在生产构建中隐藏 —— 通过 `process.env.NODE_ENV !== 'production'` 或等效检查控制，以便意外的原型合并不会将栏推送给用户。

将切换器放在单个共享组件中以便两种子形状都能复用。放在项目中共享 UI 的位置。

### 5. 交给用户

展示 URL（和 `?variant=` 键）。用户会在他们方便时浏览。有趣的反馈通常是 **"我想要 B 的头部配 C 的侧边栏"** —— 那才是他们真正想要的设计。

### 6. 捕获答案并清理

一旦一个变体获胜，写下哪个以及为什么（commit 消息、ADR、issue，或如果 AFK 运行且用户尚未回复则放在原型旁边的 `NOTES.md`）。然后：

- **子形状 A** —— 删除失败的变体和切换器；将获胜者折叠到现有页面中。
- **子形状 B** —— 将获胜的变体提升为真实路由，删除一次性路由和切换器。

不要让变体组件或切换器散落。它们很快腐烂并困惑下一个读者。

## 反模式

- **仅在颜色或文案上不同的变体。** 那是微调，不是原型。真正的变体在结构上有分歧。
- **变体之间共享太多代码。** 共享 `<Header>` 可以；共享 `<Layout>` 就违背了目的。每个变体应自由抛弃布局。
- **将变体连接到真实 mutation。** 只读原型完全可以。如果变体需要 mutation，将其指向 stub —— 问题是"这应该长什么样"，而非"后端是否工作"。
- **将原型直接提升为生产。** 变体代码是在原型约束下编写的（无测试、最小错误处理）。在折叠时正确地重写它。
