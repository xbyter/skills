# 逻辑原型

一个小型交互式终端应用，让用户手动驱动状态模型。当问题关于**业务逻辑、状态转换或数据形状**时使用 —— 那种在纸上看起来合理但只有在推动真实案例时才感觉不对的东西。

## 何时这是正确的形状

- "我不确定这个状态机是否处理了 X 然后 Y 的边缘情况。"
- "这个数据模型是否真的能表示...的情况"
- "我想在编写之前感受一下 API 应该是什么样的。"
- 任何用户想要**按按钮并观察状态变化**的场景。

如果问题是"这应该长什么样" —— 错误的分支。使用 [UI.md](UI.md)。

## 流程

### 1. 陈述问题

在编写代码之前，写下你在原型化什么状态模型和什么问题。一段话，放在原型的 README 或文件顶部的注释中。回答错误问题的逻辑原型纯粹是浪费 —— 使问题明确，以便以后可以检查，无论用户现在在看还是 AFK 后返回。

### 2. 选择语言

使用宿主项目使用的语言。如果项目没有明显的运行时（例如文档仓库），就询问。

匹配项目现有的工具约定 —— 不要仅为原型添加新的包管理器或运行时。

### 3. 将逻辑隔离在可移植的模块中

将实际逻辑 —— 回答问题的那部分 —— 放在一个小而纯的接口后面，以后可以提取并放入真实代码库。周围的 TUI 是一次性的；逻辑模块不应该。

正确的形状取决于问题：

- **纯 reducer** —— `(state, action) => state`。当动作是离散事件且状态是单个值时适合。
- **状态机** —— 显式状态和转换。当"当前哪些动作是合法的"是问题的一部分时适合。
- **一组纯函数**作用于简单数据类型。当没有隐式的当前状态时适合 —— 只有转换。
- **具有清晰方法表面的类或模块**，当逻辑确实拥有持续的内部状态时。

选择最适合所问问题的形状，*而非*最容易连接到 TUI 的。保持纯：无 I/O、无终端代码、无 `console.log` 用于控制流。TUI 导入它并调用它；没有反向流动。

这就是使原型在其生命周期之后仍有用的原因。当问题被回答后，验证过的 reducer / 状态机 / 函数集可以被提取到真实模块 —— TUI 壳被删除。

### 4. 构建暴露状态的最小 TUI

构建为**轻量级 TUI** —— 每个 tick，清屏（`console.clear()` / `print("\033[2J\033[H")` / 等效方法）并重新渲染整个帧。用户应始终看到一个稳定的视图，而非不断增长的滚动回滚。

每帧有两部分，按此顺序：

1. **当前状态**，漂亮打印且 diff 友好（每行一个字段，或格式化的 JSON）。用**粗体**表示字段名或章节标题，用**暗色**表示不太重要的上下文（时间戳、ID、派生值）。原生 ANSI 转义码就行 —— `\x1b[1m` 粗体，`\x1b[2m` 暗色，`\x1b[0m` 重置。不需要引入样式库除非项目中已有。
2. **键盘快捷键**，列在底部：`[a] add user  [d] delete user  [t] tick clock  [q] quit`。粗体键名，暗色描述，或反过来 —— 怎么清晰怎么来。

行为：

1. **初始化状态** —— 单个内存对象/结构。启动时渲染第一帧。
2. **一次读取一个按键（或一行）**，分派到改变状态的处理程序。
3. **重新渲染**每个动作后的完整帧 —— 不要追加，替换。
4. **循环直到退出。**

整个帧应适合一屏。

### 5. 使其一键可运行

在项目的现有任务运行器中添加脚本（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户应能运行 `pnpm run <prototype-name>` 或等效命令 —— 永远不需要记住路径。

如果宿主项目没有任务运行器，只需将命令放在原型 README 的顶部。

### 6. 交给用户

给用户运行命令。他们会自己驱动；有趣的时刻是当他们说"等等，那不应该是可能的"或"嗯，我以为 X 会不同"时 —— 那些是*想法*中的 bug，这正是目的所在。如果他们想添加新动作，就添加。原型会演化。

### 7. 捕获答案

当原型完成其工作后，问题的答案是唯一值得保留的。如果用户在，问它教会了他们什么。如果不在，在原型旁边留一个 `NOTES.md`，以便答案可以被填入（或者如果你观察了会话则由你填入），然后在原型被删除之前。

## 反模式

- **不要添加测试。** 需要测试的原型不再是原型。
- **不要连接到真实数据库。** 使用内存存储，除非问题专门关于持久化。
- **不要泛化。** 不要"如果我们以后想支持 X 怎么办"。原型回答一个问题。
- **不要模糊逻辑和 TUI。** 如果 reducer / 状态机引用了 `console.log`、提示符或终端转义码，它就不再可移植。保持 TUI 作为纯模块上的薄壳。
- **不要将 TUI 壳投入生产。** 壳是为从终端手动驱动而优化的。它后面的逻辑模块才是值得保留的部分。
