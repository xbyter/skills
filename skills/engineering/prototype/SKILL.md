# 原型

原型是**回答一个问题的一次性代码**。问题决定形状。

## 选择分支

识别正在回答的问题 —— 从用户的提示、周围的代码，或在用户在的时候询问：

- **"这个逻辑/状态模型感觉对吗？"** → [LOGIC.md](LOGIC.md)。构建一个小型交互式终端应用，推动状态机通过纸上难以推理的案例。
- **"这应该长什么样？"** → [UI.md](UI.md)。在单个路由上生成多种截然不同的 UI 变体，通过 URL search param 和浮动底部栏切换。

两个分支产生非常不同的产物 —— 搞错了就浪费整个原型。如果问题确实模糊且用户无法联系，默认选择更好地匹配周围代码的分支（后端模块 → logic；页面或组件 → UI），并在原型顶部陈述假设。

## 适用于两者的规则

1. **从第一天起就是一次性的，并明确标记为如此。** 将原型代码放在它实际将被使用的地方附近（在它为之原型化的模块或页面旁边），这样上下文显而易见 —— 但命名要使普通读者能看出它是原型而非生产代码。对于一次性 UI 路由，遵循项目已有的路由约定；不要发明新的顶层结构。
2. **一键运行。** 无论项目现有的任务运行器支持什么 —— `pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能不假思索地启动它。
3. **默认无持久化。** 状态存在于内存中。持久化是原型在*检查*的东西，而非它应该依赖的东西。如果问题明确涉及数据库，使用临时 DB 或有明确"PROTOTYPE — wipe me"名称的本地文件。
4. **跳过润色。** 无测试，无超出使原型*可运行*所需的错误处理，无抽象。重点是快速学到东西然后删除它。
5. **展示状态。** 每个动作后（logic）或每次变体切换后（UI），打印或渲染完整的当前相关状态，以便用户看到什么改变了。
6. **完成后删除或吸收。** 当原型回答了其问题后，要么删除它，要么将验证过的决策折叠到真实代码中 —— 不要让它在仓库中腐烂。

## 完成时

*答案*是原型中唯一值得保留的东西。将其捕获在持久的地方（commit 消息、ADR、issue，或原型旁边的 `NOTES.md`），连同它回答的问题。如果用户在，该捕获是一次快速对话；如果不在，留下占位符以便他们（或你在下一次迭代中）可以在删除原型之前填入结论。
