# 深化

如何在给定依赖的情况下安全地深化一组浅模块。假设使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇 —— **module**、**interface**、**seam**、**adapter**。

## 依赖类别

评估深化候选时，对其依赖进行分类。类别决定了深化后的模块如何跨接缝测试。

### 1. 进程内

纯计算、内存状态、无 I/O。始终可深化 —— 合并模块并通过新接口直接测试。不需要 adapter。

### 2. 本地可替代

有本地测试替代品的依赖（PGLite 代替 Postgres，内存文件系统）。如果替代品存在则可深化。深化后的模块在测试套件中运行替代品进行测试。接缝是内部的；模块外部接口没有 port。

### 3. 远程但自有（Ports & Adapters）

你自己的跨网络边界的服务（微服务、内部 API）。在接缝处定义一个 **port**（接口）。深模块拥有逻辑；传输作为 **adapter** 注入。测试使用内存 adapter。生产使用 HTTP/gRPC/queue adapter。

推荐形式：*"在接缝处定义 port，为生产实现 HTTP adapter，为测试实现内存 adapter，这样逻辑位于一个深模块中，即使它跨网络部署。"*

### 4. 真正的外部（Mock）

你无法控制的第三方服务（Stripe、Twilio 等）。深化后的模块将外部依赖作为注入的 port；测试提供 mock adapter。

## 接缝纪律

- **一个 adapter 意味着假设性接缝。两个 adapter 意味着真实接缝。** 除非至少两个 adapter 是合理的（通常是生产 + 测试），否则不要引入 port。单 adapter 接缝只是间接层。
- **内部接缝 vs 外部接缝。** 深模块可以有内部接缝（实现私有，由其自己的测试使用）以及接口处的外部接缝。不要仅仅因为测试使用内部接缝就通过接口暴露它们。

## 测试策略：替换，而非分层

- 一旦在深化后模块的接口处存在测试，浅模块上的旧单元测试就变成了浪费 —— 删除它们。
- 在深化后模块的接口处编写新测试。**接口就是测试面**。
- 测试通过接口对可观察结果做断言，而非内部状态。
- 测试应在内部重构中存活 —— 它们描述行为，而非实现。如果测试在实现改变时不得不改变，它就是在测试接口之外的东西。
