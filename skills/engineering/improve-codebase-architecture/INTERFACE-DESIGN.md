# 接口设计

当用户想为选定的深化候选探索替代接口时，使用此并行 sub-agent 模式。基于 "Design It Twice"（Ousterhout）—— 你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇 —— **module**、**interface**、**seam**、**adapter**、**leverage**。

## 流程

### 1. 框定问题空间

在生成 sub-agents 之前，为用户编写关于选定候选问题空间的面向用户的解释：

- 任何新接口需要满足的约束
- 它将依赖的依赖项，以及它们属于哪个类别（参见 [DEEPENING.md](DEEPENING.md)）
- 一个粗略的说明性代码草图来落实约束 —— 不是提案，只是使约束具体化的方式

展示给用户，然后立即进入步骤 2。用户阅读和思考的同时 sub-agents 并行工作。

### 2. 生成 sub-agents

使用 Agent 工具并行生成 3 个以上 sub-agents。每个必须为深化后的模块产生一种**截然不同**的接口。

用单独的技术简报提示每个 sub-agent（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、接缝后面的内容）。简报独立于步骤 1 中面向用户的问题空间解释。给每个 agent 不同的设计约束：

- Agent 1："最小化接口 —— 最多 1-3 个入口点。最大化每个入口点的杠杆效应。"
- Agent 2："最大化灵活性 —— 支持多种用例和扩展。"
- Agent 3："针对最常见的调用者优化 —— 使默认情况变得简单。"
- Agent 4（如适用）："围绕 ports & adapters 设计跨接缝依赖。"

在简报中包含 [LANGUAGE.md](LANGUAGE.md) 词汇和 CONTEXT.md 词汇，以便每个 sub-agent 的命名与架构语言和项目领域语言一致。

每个 sub-agent 输出：

1. 接口（类型、方法、参数 —— 加上不变量、顺序、错误模式）
2. 使用示例展示调用者如何使用
3. 实现在接缝后面隐藏了什么
4. 依赖策略和 adapters（参见 [DEEPENING.md](DEEPENING.md)）
5. 权衡 —— 哪里杠杆效应高，哪里薄弱

### 3. 展示和比较

按顺序展示设计以便用户理解每个，然后用文字比较。按**深度**（接口处的杠杆效应）、**局部性**（变更集中在哪里）和**接缝位置**进行对比。

比较后，给出你自己的推荐：你认为哪个设计最强以及为什么。如果不同设计的元素能很好地组合，提出混合方案。要有主见 —— 用户想要一个强有力的建议，而非一份菜单。
