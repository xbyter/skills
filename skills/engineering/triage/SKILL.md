# Triage

通过由 triage 角色驱动的小型状态机推动项目 issue tracker 上的 issues。

在 triage 期间发布到 issue tracker 的每一条评论或 issue **必须**以此免责声明开头：

```
> *This was generated by AI during triage.*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 如何编写持久的 agent briefs
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — `.out-of-scope/` 知识库如何工作

## 角色

两个**类别**角色：

- `bug` — 某些东西坏了
- `enhancement` — 新功能或改进

五个**状态**角色：

- `needs-triage` — 维护者需要评估
- `needs-info` — 等待报告者提供更多信息
- `ready-for-agent` — 完全明确，可供 AFK agent 使用
- `ready-for-human` — 需要人工实现
- `wontfix` — 不会被处理

每个 triaged issue 应恰好携带一个类别角色和一个状态角色。如果状态角色冲突，标记并在做任何其他事情之前询问维护者。

这些是规范角色名——issue tracker 中使用的实际标签字符串可能不同。映射应已提供给你——如果没有，运行 `/setup-matt-pocock-skills`。

状态转换：未标记的 issue 通常先到 `needs-triage`；从那里移到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。`needs-info` 在报告者回复后返回 `needs-triage`。维护者可以随时覆盖——标记看起来异常的转换并在继续前询问。

## 调用

维护者调用 `/triage` 并用自然语言描述他们想要什么。解释请求并行动。示例：

- "显示所有需要我关注的事项"
- "看看 #42"
- "将 #42 移到 ready-for-agent"
- "什么已经准备好供 agents 接手？"

## 显示需要关注的事项

查询 issue tracker 并按从旧到新呈现三个桶：

1. **未标记** — 从未 triaged。
2. **`needs-triage`** — 评估进行中。
3. **`needs-info` 且自上次 triage 笔记后报告者有活动** — 需要重新评估。

显示计数和每个 issue 的一行摘要。让维护者选择。

## Triage 特定 issue

1. **收集上下文。** 阅读完整 issue（正文、评论、标签、报告者、日期）。解析之前的 triage 笔记以免重新询问已解决的问题。使用项目的领域词汇表探索代码库，尊重该区域的 ADR。读取 `.out-of-scope/*.md` 并呈现任何类似于此 issue 的之前拒绝。

2. **推荐。** 告诉维护者你的类别和状态推荐及推理，以及与 issue 相关的代码库简要摘要。等待指示。

3. **复现（仅 bug）。** 在任何 grilling 之前，尝试复现：阅读报告者的步骤，跟踪相关代码，运行测试或命令。报告发生了什么——成功复现并附代码路径、复现失败、或细节不足（强 `needs-info` 信号）。确认的复现会使 agent brief 更有说服力。

4. **Grill（如需要）。** 如果 issue 需要充实，运行 `/grill-with-docs` 会话。

5. **应用结果：**
   - `ready-for-agent` — 发布 agent brief 评论（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` — 与 agent brief 相同结构，但注明为什么不能委托（判断调用、外部访问、设计决策、手动测试）。
   - `needs-info` — 发布 triage 笔记（下方模板）。
   - `wontfix`（bug）— 礼貌解释，然后关闭。
   - `wontfix`（enhancement）— 写入 `.out-of-scope/`，从评论中链接到它，然后关闭（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` — 应用角色。如有部分进展可附可选评论。

## 快速状态覆盖

如果维护者说"将 #42 移到 ready-for-agent"，信任他们并直接应用角色。确认你将要做什么（角色更改、评论、关闭），然后行动。跳过 grilling。如果在不进行 grilling 会话的情况下移至 `ready-for-agent`，询问他们是否想编写 agent brief。

## Needs-info 模板

```markdown
## Triage Notes

**到目前为止我们已确定的：**

- 要点 1
- 要点 2

**我们仍需要你提供的 (@reporter)：**

- 问题 1
- 问题 2
```

将 grilling 期间解决的所有内容记录在"到目前为止已确定的"下，以免工作丢失。问题必须具体且可操作，不是"请提供更多信息"。

## 恢复之前的会话

如果 issue 上存在之前的 triage 笔记，阅读它们，检查报告者是否回答了任何未解决的问题，并在继续前呈现更新的情况。不要重新询问已解决的问题。
